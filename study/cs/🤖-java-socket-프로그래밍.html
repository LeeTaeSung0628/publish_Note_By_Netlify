Socker 프로그래밍이란?

  • 소켓 프로그래밍이란 네트워크를 통해 두 장치간 데이터를 주고받기 위한 핵심 기술이다.
  • 자바는 표준 라이브러리에서 소켓 API를 제공하여 TCP/UDP 기반 통신을 쉽게 구현 가능하다.

Socket의 정의

  • 소켓은 네트워크 통신의 종단점(Endpoint) 을 의미하며, IP + Port 조합을 통해 특정 프로세스를 식별하는 기술.
Socket 과 REST API의 종단점 차이

RestApi의 경우, 클라이언트가 특정 자원(resource)에 접근하기 위해 호출하는 URL를 의미한다. 즉, 자원을 식별하는 고정된 HTTP 접근 지점 이라고 생각하면된다.

GET /api/users/123
POST /api/orders

Socket 통신 에서의 종단점은 서버와 클라이언트가 지속적으로 연결을 유지하는 네트워크주소(IP + Port)이다. HTTP의 URL과 달리, 소켓은 프로세스-프로세스 간 연결 지점을 의미한다.

203.0.113.20:9000  (TCP Chat Server)
192.168.0.10:5000 (UDP Game Server)


소켓의 구성 요소

  • IP주소 : 통신할 장치의 주소
  • 포트 : 해당 장치 내 특정 프로세스 식별
  • 프로토콜 : TCP/UDP 등
  • 데이터 스트림 : 송수신 데이터 흐름

자바 소켓 프로그래밍 원리

소켓 통신은 크게 두 흐름으로 나뉜다.

서버(Server) 측 원리

  1. 포트 바인딩
  2. 클라이언트 접속 요청 대기
  3. 접속 수락 후 소켓 생성
  4. 스트림 기반 송·수신 처리
  5. 세션 종료 및 소켓 close

이를 위해 Java는 ServerSocket 클래스를 제공한다.

클라이언트(Client) 측 원리

  1. 서버 IP와 포트로 접속 요청
  2. 연결 성공 시 소켓 생성
  3. 송·수신 스트림 확보
  4. 데이터 송신/수신
  5. 종료 처리

Java는 Socket 클래스를 사용한다.


소켓 프로그래밍의 필요성


1. 실시간 데이터 처리

개념

메신저·채팅은 서버와 클라이언트가 지속적인 연결을 유지하며 메시지를 즉시 전달해야 한다.
HTTP 요청/응답 방식은 매번 연결을 새로 열기 때문에 실시간성이 떨어진다.

왜 소켓이 필요한가?

  • 연결을 한 번 맺은 뒤 지속적으로 유지(TCP 장기 연결)
  • 메시지 전송 시 지연 시간 최소화
  • 클라이언트 → 서버, 서버 → 클라이언트 양방향 통신 가능

적용 예

  • 팀 채팅(기업용 메신저)
  • 고객센터 실시간 상담
  • 앱 in 채팅(배달앱, 쇼핑몰)

2. 서버–클라이언트 구조 시스템 구축


개념

웹·앱 서비스는 보통 다음 구조를 가진다:

  • 서버 → 클라이언트로 정보 전송
  • 클라이언트 → 서버로 요청 전송

일반적 HTTP 통신 외에, 양방향 실시간 통신이 필요한 경우 소켓이 필수다.

소켓 사용 이유

  • 서버에서 클라이언트로 실시간 푸시 가능 (알림, 메시지, 이벤트)
  • HTTP Polling 대비 리소스 절감
  • 꾸준한 연결 유지

예

  • 실시간 알림(푸시 메시지)
  • 환율/주가 실시간 갱신
  • 실시간 대시보드(모니터링 시스템)

3. 내부 서버 간 통신 (Microservice Architecture)

개념

MSA에서는 여러 서비스가 분리되어 있음.
각 서비스는 다른 서비스와 네트워크로만 통신한다.

예:

  • 인증 서버
  • 주문 서버
  • 결제 서버
  • 로그/분석 서버

소켓이 필요한 이유

  • 서비스 간 고성능 RPC 필요
  • 내부 바이너리 프로토콜 사용 가능
  • REST보다 빠르고 경량화된 통신 가능

적용 기술

  • gRPC
  • Netty 기반 고성능 TCP 서버
  • 자체 프로토콜 기반 메시지 전달
