TFM2025 을 다녀온 후기이다. 간단하게 정리한 정보를 토대로 이후 더 상세하게 기술하겠다.

설명 발췌

 2025년의 끝자락,
한 해 동안 각자의 자리에서 버텨온 개발자, 디자이너, 기획자들이 한자리에 모입니다.

이번 Tech Friends Mixer는 성과를 자랑하거나 트렌드를 나열하는 컨퍼런스가 아닙니다.
대신, 우리가 진짜로 겪은 고민들 —
기획이 꼬였던 오후, 디자인이 갈피를 못 잡던 날, 릴리즈 직전의 긴장감 —
그 순간들을 나누는 자리입니다.

말 그대로, 다양한 분야의 사람들이 모여 여러가지 이야기를 나누며 공감하는 그런 자리였다. 가벼운 마음으로 참석한 모임이지만 기대보다도 더 깊은 인사이트를 얻게 되어 참 감사한 시간이었다.

트랙1 - “말 한마디가 나의 얼굴이 되고 결과를 만든다.”

  • 업무성과가 좋아지고 회사 평가가좋아지는 마법의 한마디

전 직장 영상 편집자 동료 이야기

  • ‘시켜서 했다’ → ‘내가 판단했다’
  • ‘믿지 않는 상사’ → ‘믿어주고, 도와주는 동료’
  • 회사는 ‘나 혼자 잘한다’ → ‘옆 사람도 챙긴다’
  • 상위평가를 받는 사람들의 공통점

자율성 / 관계 역량

  • 스스로 판단하기에 상대에게 뭐가 필요한지 파악하는 역량
  • “괜찮으세요?” 등 상대를 위한 작은 배려
  • 프로덕트는 모든 파트가 중요함을 배워야 한다.
  • 자신과 더불어 타인의 이익도 챙길 줄 알아야한다.
  • 서로 win win 하는 관계가 아니라면 시작도 하지 말아라.

“도와드릴까요?”

기업문화를 선도하는 방법?

  • 일에 지장 없도록 몰래몰래 기업문화 개선을 위해 노력했다.
  • 굳이 기업문화에 반하는 말을 한다기 보다, 스스로 개선해 나가는 노력을 한다.
  • 비즈니스에 해가 되지 않도록

C트랙 : CursorAI + MCP를 이용한 TDD 개발 하기

  • 기존의 레거시 프로젝트에서 TDD를 적용할 때 얻을 수 있는 인사이트

실제 레거시 프로젝트에 적용할때의 문제점

  • 비즈니스 규칙 미반영
  • 엣지 케이스 누락
  • 실제 요구사항과 불일치

이를 해결하는 방법은 MCP

  • 이후 실습...

MCP의 역할

  • 정답이 들어있는 서버
  • AI에게 도메인 규칙 전달
  • 스펙 문서 없이도 작동
  • 레거시 코드도 안전하게 개선 가능

AI에이전트를 설계할 때, 성능 좋은 에이전트 보다 양질의 데이터? 교육을 시키는 것이 중요하다.

커서를 사용하여 시연하였는데, 커서만의 장점이 있는지?

  • 제미나이나, 클러드는 CLI에서 진행되어 룰이나 코딩하기가 힘들다.
  • 이에 반해 UX관점에서 커서는 GUI를 지원하기 때문에 쉽다.
  • 커서는 가격대비 RPM관점이나 백본?이 더 효율적이고 대중적이다.

예제에서 사용한 MCP?

  • 예제 프로젝트에 대한 비즈니스 로직에 대한 정의와 JSON 형태의 데이터.

MCP서버를 추가만했지만, MCP서버에 접근해서 보라는 컨텍스트가 없었지만 어떻게 알았는지?

  • 어떠한 기준으로 접근을해서 test코드가 나왔는지?
  • ai 백본이 작은 모델이 하나가 더 붙거나, 지금 조건에서 사용할 수 있는 tool이 뭐가있는지 파악 후 백본 ai가 실행하기 된다.
  • 백본 모델자체의 고유 능력이다.
  • 지정이 필요하다면, 텍스트 유사도 방식으로 찾아가는 것이기 때문에 명확하게 지정해주어야한다.

해당 강의에 할당된 시간이 길지 않아, 더 딥하게 들어가진 못해서 아쉬움 이 남는 기분이었다. 이후 강의가 끝난 이후에 너무 감사하게도 20분 가량 시간내어 주셔서 더 깊은 대화를 할 수 있었다.

부제 - 대 AI시대의 대처하는 방법?

강의가 너무짧아서 아쉬웠다는 스몰토킹으로 시작하여 여러가지 방향에 대해 제시해주신 권도현 선생님,

  1. 얕고 넓은 지식의 중요성
  2. 검토자의 입장
  3. 성능좋은 에이전트가 아닌, 상황에 특화된 다중 에이전트와 MCP
  4. 비즈니스 지식의 중요성 개발자가 놓칠 수 있는, 또한 AI가 평생 따라오지 못할 부분 ‘비즈니스’
  5. ..

C트랙 : Exabyte-Scale 데이터를 다루는 방법

빅데이터 , 하둡 (빅데이터 플랫폼 구축)

데이터 엔지니어 직군

  • 데이터 엔지니어 - 데이터를 위한 안정적인 파이프라인 (데이터수집, ETL)을 설계,구축,관리
  • 데이터 분석가 - 데이터를 분석하여 비즈니스 인사이트를 도출, 시각화 , 보고서
  • 데이터 사이언티스트 - 통계, 수학, 프로그래밍 지식을 활용해 데이터 분석

빅데이터란?

  • 기존 DB로 처리하기 힘든 대용량 데이터 -하나의 서버에서 저장 및 처기 불가능

데이터 예시

  • 로그 원본데이터
  • 사용자 행동 데이터
  • 웹 크롤링 데이터

활용예시

  • FDS(이상거래탐지시스템)
  • 추천 시스템 - 광고추천 등 , 개인화 모델
  • AI모델 학습

빅데이터 플랫폼?

데이터 웨어하우스

  • 굉장히 큰 데이터를 다룰 수 있는 SQL 플랫폼?

데이터 레이크

  • 커다란 저장공간을 제공한다.
  • 컴퓨팅을 따로 제공하지는 않으며, 텍스트위주의 큰 데이터를 제공

프레임워크 / 제품

  • 하둡 / 구글 빅 쿼리 / 아마존 EMR 등등.. 이 해당된다.

카카오는 하둡 기반으로 자체 구축(on-premise)

  • 카카오의 데이터 플랫폼 단위는 EB단위. ( 1EB - 1000PB - 1,000,000TB )
  • AWS S3 standard 기준으로 1EB당 공간비용은 월 360억
  • 이와같이 일정 수준의 범위를 넘어가면 외부플랫폼 비용이 증가하기 떄문에, on-premise환경으로 구축한다.

HDFS ( Hadoop Distributed File System)

  • 하둡의 기본 스토리지 레이어
  • 대규모 데이터를 클러스터 내 여러 노드에 분산하여 저장
    • 데이터 복제본을 여러벌 저장하여 고가용성 확보
  • 데이터는불변으로, 쓰기 완료 후 일부 수정 불가능
  • 높은 처리량에 최적화

1015데이터 화재로 인하여 IDC(데이터센터) 3중화를 진행하게 되었다.(기존데비 용량 증가 없음)

레플리카 방식과 EC(erasure coding)의 차이

  • 3레플리카 대신 EC를 사용한다면 저정용량이 크게 줄어든다.

Apach ozone 기반 대규모 분산 오브젝트 스토리지 운영
